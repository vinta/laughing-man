# laughing-man: Markdown-to-Newsletter CLI Tool

## Overview

A CLI tool that turns a folder of markdown documents into a newsletter: a static archive website on Cloudflare Pages and email delivery via Resend Broadcasts. Built with Bun + TypeScript. Designed for open source from day one.

**Repository:** Standalone repo (not inside of-course-i-still-love-you).

**Runtime:** Requires Bun. Distributed as npm package, invoked via `bunx laughing-man`.

## CLI Interface

```bash
laughing-man init              # Generate laughing-man.yaml
laughing-man build             # Validate + generate site + email HTML to output/
laughing-man preview           # Build + start local server for preview
laughing-man deploy            # Deploy output/website/ to Cloudflare Pages
laughing-man send 1            # Send issue 1 via Resend Broadcast
laughing-man send 1 --yes      # Non-interactive mode (for CI)
```

`build`, `deploy`, and `send` are separate, composable steps. No monolithic `publish` command. Run them independently or chain them in CI.

## Config

Generated by `laughing-man init`. Lives in the user's newsletter directory as `laughing-man.yaml`.

```yaml
name: "The Net is Vast and Infinite"
url: "https://thenetisvastandinfinite.com"

issues_dir: .
attachments_dir: ../Attachments

web_hosting:
  provider: cloudflare-pages
  project: laughing-man              # Cloudflare Pages project name

email_hosting:
  from: "Vinta <vinta@thenetisvastandinfinite.com>"
  reply_to: vinta@thenetisvastandinfinite.com
  provider: resend

env:
  resend_api_key: "re_xxxxx"              # or RESEND_API_KEY from environment variables
  resend_audience_id: "aud_xxxxx"         # or RESEND_AUDIENCE_ID from environment variables
```

**Environment variable override:** Values in the `env` section can be overridden by environment variables. `RESEND_API_KEY` overrides `resend_api_key`, etc. The tool loads `.env` from the newsletter directory if it exists, so locally you keep secrets in `.env` (gitignored) and in CI inject them as real env vars.

Cloudflare credentials are not in the config. Wrangler handles its own auth: locally via `wrangler login` (OAuth), in CI via `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` env vars passed directly to the workflow.

A `laughing-man.example.yaml` ships with the package for open source reference.

## Content Format

Each newsletter issue is a markdown file with YAML frontmatter:

```markdown
---
issue: 1
status: ready
---

# 創刊號：網路是無限寬廣的！

Content here in markdown...
```

### Frontmatter fields

| Field    | Required | Default | Purpose                                             |
| -------- | -------- | ------- | --------------------------------------------------- |
| `issue`  | yes      | -       | Issue number, determines URL and sort order         |
| `status` | yes      | -       | `draft` or `ready`. Controls visibility (see below) |

**Title** is extracted from the first `# heading` in the markdown body. No frontmatter needed.

Filenames are arbitrary. The tool discovers issues by scanning `issues_dir` for `.md` files and reading their frontmatter.

### Validation rules

- Files missing `issue` or `status` produce a **build error** with the filename.
- Files with invalid `status` (not `draft` or `ready`) produce a **build error**.
- Files missing a `# heading` produce a **build error**.
- Duplicate `issue` numbers across files produce a **build error** listing both files.

### Status field

- `draft`: included in `preview`, excluded from `build`/`deploy`/`send`. Work in progress.
- `ready`: included everywhere. Ready to publish.

## Images

Markdown files can reference images with relative paths:

```markdown
![](Attachments/the-net-is-vast-and-infinite.jpg)
```

### Image resolution

The tool resolves image paths in this order:

1. Relative to the markdown file's location
2. Relative to `attachments_dir` (if configured)

This supports Obsidian-style vaults where `Attachments/` is a shared folder outside the issues directory.

### Build-time image handling

1. Scan rendered HTML for `<img>` tags with relative `src` attributes
2. Resolve the source file using the resolution order above
3. Copy referenced files to `output/website/images/<issue>/`
4. Rewrite `src` in **web HTML** to site-relative paths (`/images/1/the-net-is-vast-and-infinite.jpg`)
5. Rewrite `src` in **email HTML** to absolute URLs (`https://thenetisvastandinfinite.com/images/1/the-net-is-vast-and-infinite.jpg`) using `config.url`

Email clients do not support relative image paths. All email images must be absolute URLs.

If a referenced image cannot be found in any resolution path, the build fails with an error naming the file and the markdown source.

## Theme System

### Built-in themes

Themes live inside the `laughing-man` package source. Users do not interact with template files directly.

```
laughing-man (package source)/
  themes/
    default/
      email.ts           # Email HTML template (plain function)
      web.ts             # Template for individual issue page
      index.ts           # Archive/home page listing all issues
      styles.css         # Default styles
```

All templates are plain TypeScript functions returning HTML strings. No React or JSX. v1 ships with one theme: `default`. No customization in v1.

### Template props

Templates receive issue metadata and rendered content:

```typescript
interface IssueProps {
  title: string; // Extracted from first # heading
  issue: number;
  content: string; // Rendered HTML from markdown
  config: SiteConfig;
}
```

## Build Pipeline

`laughing-man build` runs this pipeline:

1. Read `laughing-man.yaml` (with env var overrides applied)
2. Scan `issues_dir` for `.md` files, parse frontmatter with gray-matter
3. Validate frontmatter: fail on missing required fields, duplicate issue numbers
4. Filter out `status: draft` issues
5. Sort issues by issue number
6. For each issue:
   - Render markdown to HTML (via marked)
   - Process images: resolve relative paths, copy to output, rewrite URLs
   - Render email HTML via theme's `email.ts` (absolute image URLs)
   - Render web HTML via theme's `web.ts` (site-relative image URLs)
7. Render `index.ts` with the list of all issues
8. Write everything to `output/website/` and `output/email/`

## Preview

`laughing-man preview` builds and starts a local HTTP server via `Bun.serve`:

- Serves `output/website/` on `localhost:4000` (or next available port)
- Opens the browser automatically
- Includes `status: draft` issues so you can preview work in progress

## Deploy

`laughing-man deploy` uploads `output/website/` to Cloudflare Pages via `wrangler pages deploy`.

Requires `wrangler` to be installed (`bun add -D wrangler` or globally). Wrangler handles its own authentication:
- **Locally:** Run `wrangler login` once. OAuth token is stored and auto-refreshed.
- **In CI:** Set `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` as env vars.

Requires `build` to have been run first. If `output/website/` does not exist, errors with a message to run `build`.

## Send

`laughing-man send <issue-number>` sends an issue via Resend Broadcasts:

1. Reads `output/email/<issue>.html`
2. Creates a Resend Broadcast targeting the configured audience
3. Resend handles delivery, unsubscribe links, and suppression natively

### Safety checks

- **Duplicate send prevention:** Queries the Resend API for existing broadcasts matching this issue number. If found, warns and aborts. Works across CI runs and local machines with no local state file.
- **Status check:** Refuses to send issues with `status: draft`.
- **Build check:** Requires `output/email/<issue>.html` to exist. Errors with a message to run `build` first.
- **Confirmation prompt:** Shows "Send issue 1 'title' to audience? [y/N]" before sending. Skipped with `--yes` for CI.

## Subscribe

The archive site includes a subscribe form on the index page. Visitors enter their email and are added to the Resend audience via a Cloudflare Pages Function.

### How it works

1. The index page includes an email input + submit button
2. Form submits via `fetch()` to `POST /api/subscribe` with `{ "email": "..." }`
3. `/api/subscribe` is a Cloudflare Pages Function (`functions/api/subscribe.ts`) that:
   - Validates the email format
   - Calls `POST https://api.resend.com/contacts` with the email and audience ID
   - Returns JSON `{ "ok": true }` or `{ "error": "..." }`
4. The form shows inline success/error feedback, no page reload

### Single opt-in

Contacts are added directly to the audience in an active state. No confirmation email. This keeps the flow simple and maximizes conversion. Double opt-in can be added later by creating contacts with `unsubscribed: true` and flipping the flag after email verification.

### Pages Function environment

The Pages Function reads `RESEND_API_KEY` and `RESEND_AUDIENCE_ID` from Cloudflare Pages environment variables. These are set via the Cloudflare dashboard (Settings > Environment variables) or `wrangler pages secret put`. They are the same credentials used by the `send` command, but configured separately for the Pages runtime.

The function uses `fetch()` directly against Resend's REST API. No SDK needed at the edge.

### No new dependencies

The Pages Function runs in the Cloudflare Workers runtime, not in the npm package. It uses the Web `fetch()` API to call Resend, so no `resend` SDK import is needed. The function file ships as part of the build output.

## CI Support

Sample GitHub Actions workflow (documented in README, not generated by the tool):

```yaml
name: Publish Newsletter
on:
  workflow_dispatch:
    inputs:
      issue:
        description: "Issue number to send"
        required: true

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - run: bunx laughing-man build
      - name: Deploy to Cloudflare Pages
        uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          command: pages deploy output/website --project-name=laughing-man

  send:
    needs: build-and-deploy
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - run: bunx laughing-man build
      - run: bunx laughing-man send ${{ inputs.issue }} --yes
        env:
          RESEND_API_KEY: ${{ secrets.RESEND_API_KEY }}
          RESEND_AUDIENCE_ID: ${{ secrets.RESEND_AUDIENCE_ID }}
```

Deployment uses Cloudflare's `wrangler-action` to upload `output/website/` directly to Cloudflare Pages via direct upload. The Cloudflare Pages project must be created first (Workers & Pages > Create > Pages > Upload assets). No `gh-pages` branch needed; no GitHub Pages configuration required. Works with private repos on the free plan. Each job is independent and can be retried individually.

**Why Cloudflare Pages over GitHub Pages:**
- Works with private repos on the free GitHub plan (GitHub Pages requires Pro for private repos)
- Unlimited bandwidth (GitHub Pages has a 100GB/mo soft cap)
- Global edge network with automatic asset optimization
- Aligns with the Cloudflare Workers stack planned for the subscribe system in v2

## Key Dependencies

- **@11ty/gray-matter** - YAML frontmatter parsing (actively maintained fork of gray-matter)
- **marked** - Markdown to HTML
- **resend** - Resend API SDK (Broadcasts)
- **zod** - Config and frontmatter validation

Config YAML parsing uses `Bun.YAML.parse()` (built-in, no dependency). Email templates are plain HTML functions (no React dependency).

## Open Source Design Principles

- Provider logic isolated in `src/providers/`. Adding a new email or hosting provider means adding a file, not modifying existing ones.
- No user data in the package. All state lives in the user's directory or the provider (Resend).
- Config-driven. No hardcoded assumptions about newsletter name, domain, or provider details.
- No local state files for distributed concerns. Send state lives in Resend API, deploy state lives in Cloudflare Pages.

## Out of Scope for v1

- Theme customization (config tokens, CSS overrides, user-authored templates)
- Multi-language support
- Binary distribution (`bun build --compile`)
- `eject-theme` command
- Paid tiers or monetization
- Analytics or open/click tracking
- Double opt-in confirmation flow
- Webhook processing for delivery events
- Multiple newsletter support
- Live reload in preview
- CAPTCHA / bot protection (can add Cloudflare Turnstile later)
- Welcome email on subscribe
